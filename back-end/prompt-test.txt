            - Chunking Efficiency (Hiệu quả chunking).
        3. Alerting: Thiết lập ngưỡng cảnh báo để phát hiện vấn đề kịp thời.
    C. Error Handling (Xử lý lỗi):
        1. Mục tiêu: Giảm thiểu tác động khi hệ thống gặp sự cố (graceful degradation).
        2. Chiến lược Fallback:
            - Nếu RAG pipeline lỗi -> Thử Keyword Search.
            - Nếu Keyword Search lỗi -> Trả về các chunk đã truy xuất.
            - Nếu vẫn lỗi -> Sử dụng Text Matching đơn giản.
            - Cuối cùng: Trả về thông báo lỗi hữu ích.
        3. Half-open State: Kiểm tra định kỳ để khôi phục hoạt động bình thường.
    D. Kiến trúc hệ thống RAG trong sản xuất:
        1. Layered Architecture: Data Layer, RAG Pipeline Layer (microservices), Application Layer, Monitoring Stack.
        2. Công nghệ: Kubernetes, ChromaDB, Redis, PostgreSQL, Prometheus, Grafana, Jaeger, ELK stack.
        3. Lợi ích: Khả năng mở rộng, tính sẵn sàng cao, hiệu suất.
XI. Model Context Protocol (MCP)
    A. Vấn đề: LLM chỉ có thể tạo sinh văn bản/nội dung, không thể thực hiện hành động hoặc tương tác với các hệ thống bên ngoài.
    B. AI Agents:
        1. Khái niệm: Hệ thống có khả năng tương tác với các nền tảng bên thứ ba, thu thập thông tin, sử dụng bộ nhớ và tương tác với LLM để đưa ra quyết định và thực hiện hành động.
        2. So sánh với Automation Scripts cũ: AI Agents "có thể suy nghĩ" nhờ LLM.
        3. Quy trình làm việc điển hình: Yêu cầu người dùng -> Tương tác với LLM để trích xuất chi tiết -> Tương tác với LLM để xác định công cụ cần dùng -> Tương tác với công cụ (API bên thứ ba) -> Tương tác với LLM để ra quyết định -> Thực hiện hành động.
    C. Tools (Công cụ):
        1. Khái niệm: Cho phép AI Agent tương tác với các nền tảng khác.
        2. Vấn đề: Mỗi API bên thứ ba có định dạng và cách thức tương tác khác nhau, đòi hỏi viết code riêng cho từng API.
    D. MCP (Model Context Protocol):
        1. Khái niệm: Một bộ tiêu chuẩn định nghĩa cách các ứng dụng AI có thể làm việc với nhau, cung cấp ngữ cảnh cho AI Agent để tương tác với các nền tảng bên thứ ba.
        2. Mục tiêu: Giúp AI Agent khám phá và sử dụng các API một cách chuẩn hóa.
        3. Kiến trúc Client-Server:
            - MCP Client: Thường được nhúng trong IDEs (Cursor, VS Code extensions) hoặc các ứng dụng AI.
            - MCP Server: Cung cấp các khả năng (resources, prompts, tools) cho Client.
        4. Ai xây dựng MCP Server? Chủ sở hữu ứng dụng/dịch vụ hoặc cộng đồng.
    E. Các thành phần của MCP:
        1. Specification: Bộ quy tắc định nghĩa giao tiếp (JSON RPC, stateful connection).
        2. Resources: Dữ liệu mà Server cung cấp (ví dụ: danh sách sân bay, chính sách hoàn tiền). Có URI, tên, mô tả.
        3. Tools: Các hành động mà Server có thể thực hiện (ví dụ: tìm chuyến bay, đặt vé). Có mô tả, schema đầu vào/đầu ra.
        4. Prompts: Các mẫu prompt được định nghĩa sẵn để hướng dẫn LLM. Có tên, mô tả, tham số.
    F. Giao tiếp Client-Server:
        1. JSON RPC: Giao thức gọi thủ tục từ xa, định nghĩa cách Client gọi phương thức trên Server và nhận phản hồi.
        2. Transport Mechanisms: HTTP, Standard IO (cho kết nối cục bộ).
    G. Hands-on Lab 6: Sử dụng MCP Server có sẵn (Root Code AI assistant).
        1. Cấu hình API keys.
        2. Kết nối với MCP Server (flight booking server).
        3. Kiểm tra các Tools và Resources.
        4. Sử dụng Root Code để tương tác với MCP Server.
    H. Xây dựng MCP Server:
        1. Sử dụng SDKs (ví dụ: Python SDK).
        2. Định nghĩa Resources, Tools, Prompts dưới dạng hàm Python với decorator `@mcp.resource`, `@mcp.tool`, `@mcp.prompt`.
        3. Chạy Server với các tùy chọn transport (Standard IO, HTTP).
    I. Hands-on Lab 7: Xây dựng MCP Server.
        1. Thiết lập dự án Python.
        2. Thêm decorator cho Resources, Tools, Prompts.
        3. Cấu hình MCP Server cho Root Code Agent.
    J. Xây dựng MCP Client:
        1. Các tính năng Client:
            - Roots: Thư mục mà Server được phép truy cập trên máy Client.
            - Sampling: Client kiểm soát việc LLM được Server gọi (chọn mô hình, giới hạn token).
            - Elicitation: Client lấy thông tin từ người dùng cuối để Server sử dụng.
            - Context: Server gửi cập nhật tiến trình/thông tin về Client.
        2. Sử dụng thư viện `fast_mcp_client`.
    K. Hands-on Lab 8: Xây dựng MCP Client.
        1. Khám phá các khái niệm Client (Roots, Sampling, Elicitation, Context).
        2. Thực hành xây dựng Client.
XII. Tổng kết
    A. RAG: Truy xuất thông tin động để cải thiện chất lượng phản hồi của LLM.
    B. MCP: Chuẩn hóa cách AI Agent tương tác với các hệ thống bên ngoài.
    C. Kết hợp RAG và MCP: Xây dựng các hệ thống AI phức tạp, thông minh và có khả năng hành động.
